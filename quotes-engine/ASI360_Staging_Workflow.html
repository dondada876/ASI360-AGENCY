ASI 360 — Staging Environment

Safe front-end development workflow so DMX never breaks production again
The Problem
No Staging = Production Goes Down When Something Breaks

Right now, changes to asi360.co happen live. If DMX (or anyone) pushes a CSS change that breaks the header, or a JS error that kills the page, customers see it immediately. There's no "undo" button and no safe place to test. The solution is a staging environment — a copy of the site where you can break things freely, and only push to production when it's confirmed working.

What happened last time: Changes went directly to production → something broke → site went down → had to scramble to fix. This workflow eliminates that risk entirely.
Three Options
SiteGround Staging vs. Subdomain Clone vs. GitHub + Local Dev

Option A — SiteGround Built-In Staging (Fastest, Easiest, Free)

SiteGround has a one-click staging tool built into Site Tools. It creates an exact copy of your WordPress site at a staging URL. DMX works on the staging copy. When everything looks good, you push staging → production with one click. If anything breaks on staging, production is untouched.

One-click create: exact clone of asi360.co with all plugins, themes, content
Free — included in your SiteGround plan
Same server, same PHP, same environment — no "works on my machine" issues
One-click merge staging → production when ready
Password-protected by default — customers can't see the staging site
DMX gets full wp-admin access on staging without touching production
Can push just files, just database, or both
Only 1 staging copy at a time (can't have 2 designers on 2 different versions)
No Git history — changes aren't version-controlled
This is the move for right now. SiteGround staging takes 5 minutes to set up, costs nothing, and gives DMX a safe playground. You don't need Git or a complex CI/CD pipeline for front-end design work. Just click "Create Staging" and hand DMX the login.

Option B — Manual Subdomain Clone (dev.asi360.co)

Create a subdomain like dev.asi360.co, install a fresh WordPress copy, import the theme/plugins/content, and use it as a permanent dev environment. More control than SiteGround staging but more setup work.

Permanent dev environment — doesn't need to be recreated each time
Can run different plugins/themes for experimentation
Separate database — impossible to corrupt production
Manual setup: install WP, activate licenses, import content
Uses one of your Astra/Spectra license activations (3/1000 — fine)
No one-click merge — changes must be manually replicated to production
Content drift: staging gets out of sync with production over time

Option C — GitHub Repo + Local Dev + Deploy Pipeline (Pro Setup)

Export the theme files and custom code to a GitHub repo. DMX clones locally, makes changes, pushes to GitHub. A deploy script syncs approved changes to production via FTP through the droplet.

Full Git version control — every change tracked, reviewable, reversible
Multiple designers can work on branches simultaneously
Code review before anything touches production
Professional workflow — scales to a team
DMX needs to run WordPress locally (Local by Flywheel, MAMP, Docker)
Setup time: 2-4 hours to get everything working
Only covers theme files + custom CSS/JS — not Gutenberg block content
Spectra block designs live in the database, not in files — Git can't track them
Overkill for front-end design tweaks

Setup Guide
Option A — SiteGround Staging in 5 Minutes

Log into SiteGround Site Tools

Go to tools.siteground.com → select asi360.co → Site Tools dashboard

Create Staging Copy

Navigate to WordPress → Staging in the left sidebar. Click "Create Staging". SiteGround will clone the entire site — files, database, plugins, themes, everything — to a staging URL like stg-asi360co.siteground.cloud

This takes 1-3 minutes depending on site size.

Give DMX Staging Credentials

The staging site has its own wp-admin. Either:

a) Share the staging URL + existing admin credentials (they work on staging too), OR

b) Log into staging wp-admin → Users → Add New → create a "dmx-designer" account with Editor or Administrator role

DMX can now go wild on staging. Nothing touches production.

DMX Makes All Changes on Staging

DMX works exclusively on stg-asi360co.siteground.cloud:

— Customizer changes (colors, fonts, header, footer)

— Page edits with Spectra blocks

— New pages, new layouts

— CSS customizations

— Menu changes, widget changes

— Everything.

You Review Staging Before Pushing Live

Visit the staging URL yourself. Check every page. Make sure nothing is broken. Ask DMX to walk you through the changes. Only proceed when you're satisfied.

Push Staging → Production (One Click)

Back in SiteGround Site Tools → WordPress → Staging → click "Push to Live".

Choose what to push:

— Full Push: Files + Database (replaces everything)

— Files Only: Theme, plugins, CSS/JS (keeps production content/orders)

— Database Only: Content, pages, settings (keeps production files)

For design changes: "Full Push" is usually correct. For content-only updates: "Database Only".

Verify Production

Visit asi360.co and confirm everything looks right. SiteGround automatically creates a backup before pushing, so if anything goes wrong you can restore in 1 click under Site Tools → Security → Backups.

Important: While staging exists, don't make changes on BOTH staging and production simultaneously. That creates merge conflicts. The rule is: all changes happen on staging, and production only gets updated via the "Push to Live" button.

Optional: Add Git Later
Phase 2 — GitHub Repo for Theme Customizations

Once the staging workflow is running smoothly, you can layer on a GitHub repo for tracking custom theme code (child theme CSS, custom JS, mu-plugins). This doesn't replace SiteGround staging — it adds version control on top.

GitHub Repo: asi360/asi360-theme
├── style.css — Child theme stylesheet
├── functions.php — Custom PHP hooks (careful!)
├── custom.css — Additional CSS overrides
├── custom.js — Front-end scripts
├── mu-plugins/ — Must-use plugins (mega menu, custom post types)
├── assets/ — Custom icons, images, fonts
├── .github/workflows/deploy.yml — Auto-deploy on merge to main
└── README.md — Setup instructions for designers

DMX Workflow with Git:
1. git checkout -b feature/new-header
2. Make CSS/JS changes locally
3. git push → create Pull Request
4. Don reviews PR → approves → merges to main
5. GitHub Action → FTP deploys to SiteGround via droplet
6. Only theme files deploy — page content stays in WordPress

IMPORTANT: Spectra block content (page layouts, sections) lives
in the WordPress database, NOT in files. Git only tracks the theme
layer (CSS, JS, PHP). For page design changes, use SiteGround staging.
When to add Git: When DMX is writing custom CSS/JS files that you want version-controlled. Not needed for drag-and-drop Spectra block design — that's what staging is for. The two tools serve different purposes: staging = safe content/design changes, Git = tracked code changes.
